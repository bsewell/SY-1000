## Changes Made This Session

### 1. `src/customParaEQGraph.cpp` — PEQ drag crash (commit 4a3f8fe)

Two root causes for Colin's "crash when dragging PEQ node out of range":

**Fix 1 — Node position detection (paintEvent, lines ~407-410)**
Was using a strict pixel threshold to find the closest sample to each mid-band
centre. If the centre fell exactly between two sample points, the node stayed at
`QPointF(0,0)`, producing spurious hit-tests at the top-left corner of the
widget.
Changed to closest-sample tracking (`isNull()` seed + distance comparison).

**Fix 2 — mouseMoveEvent coordinate clamping (lines ~442-452)**
Raw `pMouseEvent->pos()` was passed directly to `dragNode` with no bounds check.
On Windows, dragging outside the widget edge produces coordinates outside the
plot area, corrupting stored node positions.
Fixed: clamp to `[0, width-1] × [0, height-1]` before calling `dragNode`.

---

### 2. `src/customParaEQGraph.cpp` — Ollama review defensive fix (commit c3031fa)

**updateSlot hex validation (lines ~537-563)**
All 11 `toUShort(&ok, 16)` calls shared a single `bool ok` that was never
checked. A corrupt or truncated SysEx packet could silently pass 0 to `set*()`
calls, leaving node positions in an inconsistent state that could retrigger the
null-node crash.
Fixed: validate all 11 conversions up front; bail early on any failure before
touching any node values.

---

### 3. `src/main.cpp` — Ollama review defensive fix (commit c3031fa)

**initCrashLog mkdir guard (lines ~420-448)**
`mkdir()` return values were silently ignored. If the log directory couldn't be
created, `writeRawLog` would try to open a file in a non-existent directory
(logs silently lost, including crash data).
Fixed:
- Added `#include <cerrno>` for `EEXIST` check
- Check the final `mkdir(dirPath, ...)` result; if it fails with anything other
  than `EEXIST`, fall back to `/tmp` (POSIX) or `C:\Temp` (Windows)
- Log a `WARNING:` line once the log file is open so the fallback is visible

### 4. `src/customParaEQGraph.cpp` — Mid freq overlap + high cut direction (commits 3dc2bf7, 64a5065)

**Mid freq overlap (Colin feedback — commit 3dc2bf7)**
LowMid was locked to 20–38% of plot width; HighMid to 58–76%. The zones never
overlapped, so you couldn't drag both mid nodes to the same frequency even
though the SY-1000 hardware allows it.
Fixed: both nodes now share the full mid zone 20–76% (`* 0.56` instead of
`* 0.18`). Drag-to-frequency mapping updated to match.

**High cut backwards (Colin feedback — initial wrong fix 3dc2bf7, correct fix 64a5065)**
The first attempt just changed the X direction. After studying the MIDI
implementation spec (line 22165–22189):

```
HIGH CUT (0-30):
  value 0  = 20.0 Hz  (maximum cut)
  value 30 = FLAT     (no effect)
```

This is the **inverse** of LOW CUT (value 0=FLAT, value 30=16kHz max cut).

The original code computed `highCutAmount = m_iHighCut / 30.0`, so:
- value 0 → amount=0 → `if(amount > 0)` guard suppressed all rendering (no cut
  shown even at 20Hz = maximum cut)
- value 30 → amount=1 → maximum cut displayed (should be FLAT)

Correct fix: `highCutAmountInv = 1.0 - m_iHighCut / 30.0`. Node is now
invisible at value 30 (FLAT ✓), and at maximum depth + leftmost X position at
value 0 (20Hz ✓). The dead `highCutAmount` variable was removed.
